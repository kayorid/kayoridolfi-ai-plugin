1. Visão geral

Kayoridolfi AI transforma o Claude Code num parceiro de engenharia disciplinado: regras invioláveis, ciclo Spec-Driven obrigatório, review formal, observabilidade e segurança como hooks bloqueantes, custo auditado, evals para features de IA, e — desde v2.1 — inteligência operacional (knowledge graph, search, drift).

O problema que resolve

Assistentes de IA são poderosos e imprudentes: editam código sem confirmar, executam comandos destrutivos, deixam segredos em commits, fazem suposições arquiteturais não-revisadas, e o squad perde a memória das decisões. Kayoridolfi AI adiciona fricção exatamente nos pontos certos: bloqueio antes do dano, registro auditado, checkpoints obrigatórios.

4 princípios fundamentais

  • Stack-agnostic. O harness não opina sobre linguagem ou framework. /kai-bootstrap aprende o repo e gera contexto local; nada é hardcoded para React, Go, Python ou qualquer stack.
  • Verificação antes de claim. Nada é "pronto" sem evidência observável. Hooks BLOCK impedem ação destrutiva; agentes especializados fazem review formal; evals comparam saídas de IA contra rubricas.
  • Auditoria por padrão. Toda exceção, decisão, custo e evento crítico vai para .kai/audit/ em formato append-only. Pipe-separated logs ou JSONL — fácil de fazer grep e parse.
  • Local-first. Knowledge graph, telemetria, índices de busca, retros, audit logs — tudo no repo. Sem cloud, sem rede, sem telemetria externa. Você controla os dados.
Novidade v2.1.0 — Inteligência operacional

Plugin kai-intel adiciona knowledge graph cross-squad, busca BM25 sobre artefatos e drift detection entre spec e código. Mais: /kai-telemetry opt-in 100% local e hook cost-finalize.sh que lê transcript JSONL completo para agregar tokens reais por modelo.

2. Arquitetura

Kayoridolfi AI é um marketplace de plugins do Claude Code. Cada plugin é independente, com manifest plugin.json, comandos *.md com frontmatter, hooks declarados em hooks.json, skills auto-discoverable e scripts shell.

Anatomia de um plugin

plugins/kai-intel/ ├── .claude-plugin/ │ └── plugin.json ← manifest: name, version, description ├── README.md ← documentação do plugin ├── commands/ ← cada *.md é um /comando │ ├── kai-graph-add.md frontmatter: description, argument-hint │ ├── kai-graph-query.md │ ├── kai-graph-export.md │ ├── kai-search.md │ └── kai-drift-check.md ├── skills/ ← skills auto-discoverable │ └── kai-intel/ │ └── SKILL.md description rica = trigger de invocação ├── scripts/ ← implementação shell │ ├── graph.sh │ ├── search.sh │ └── drift-check.sh └── hooks/ ← opcional ├── hooks.json declara PreToolUse, PostToolUse, Stop, etc. └── scripts/ └── *.sh scripts invocados pelo hook engine

Fluxo de um comando

Usuário → /kai-graph-add → command .md → scripts/graph.sh add ... → .kai/intel/graph.json

O .md com frontmatter funciona como "controller": instrui o Claude a executar o script com os args, e a apresentar o resultado. Toda lógica determinística (CRUD, parsing, ranking) fica no shell.

Fluxo de um hook

Claude tenta tool → PreToolUse hooks → secret-scan.sh → PASS
tool exec → PostToolUse → cost-capture.sh → audit-log.sh → .kai/audit/*.log
fim de sessão → Stop / SessionEnd → cost-finalize.sh → lê transcript JSONL → agrega por modelo

Hooks recebem JSON via stdin (payload do evento) e respondem por exit code + stdout. Exit ≠ 0 em PreToolUse bloqueia a tool.

01 / 10 kai-ai-core
obrigatório

O núcleo. Constitution, hooks de segurança bloqueantes, MCP allowlist, dashboard, telemetria opt-in.

O que faz

Define as regras invioláveis do harness e fornece a infraestrutura compartilhada que os outros 9 plugins assumem: paleta de cores, ASCII art, lib de output, achievements, banner de SessionStart, statusline customizada, snapshot do estado do squad, doctor de auto-diagnóstico, e os hooks bloqueantes mais críticos (secret scan, MCP allowlist).

Na prática

  1. Você abre o Claude Code → banner SessionStart mostra versão, squad atual, status dos plugins.
  2. Tenta dar git commit com uma chave de API no diff → secret-scan bloqueia a operação, audit log registra a tentativa.
  3. Roda /kai-doctor → relatório completo: plugins instalados, hooks ativos, versões sincronizadas, paths esperados.
  4. Quer "rapidamente" testar algo → /kai-fast <tarefa> pula o ciclo SDD com fricção mínima (mas registra que pulou).
  5. No fim do quarter → /kai-achievements mostra o que o squad desbloqueou (16+ achievements gamificados, sem ranking competitivo).

Tecnicamente

Constitution

Documento markdown que define as 4 regras invioláveis. Hooks fazem enforcement; agentes de review citam violations. Localização: plugins/kai-ai-core/skills/kai-constitution/SKILL.md.

MCP allowlist

JSON de MCPs aprovados. Hook mcp-allowlist.sh intercepta tool calls de MCPs externos e bloqueia se o servidor não estiver na lista. Exceção via /kai-exception (auditada).

02 / 10 kai-bootstrap
opt-in

Onboarding híbrido do squad: análise automática do repo + entrevista guiada + plano de enriquecimento.

O que faz

Faz o Claude Code conhecer seu squad antes de tentar ajudar. Combina análise estática do repo (linguagens, frameworks, scripts) com entrevista interativa (domínio de negócio, prioridades, restrições) e produz um .kai/squad-context.md que vira parte do contexto de toda sessão futura.

Na prática

  1. Primeiro dia do squad com o harness instalado → /kai-bootstrap
  2. Script repo-scan.sh analisa: linguagens, frameworks, package managers, CI configs, presença de tests, README, docs.
  3. Claude pergunta: "Qual o domínio do squad? Quais squads dependem do seu output? Quais são as prioridades dos próximos 3 meses?"
  4. Gera .kai/squad-context.md + plano em .kai/enrichment-plan.md com tasks priorizadas (glossário, runbooks, skills custom).
  5. Trimestralmente: /kai-bootstrap-rescan atualiza sem perder o contexto editado à mão.

Tecnicamente

Detecção automática usa heurísticas em plugins/kai-bootstrap/skills/kai-bootstrap/scripts/repo-scan.sh:

  • Linguagem: contagem de extensões + presença de package.json, go.mod, Cargo.toml, etc.
  • Framework: grep em package.json ou imports comuns
  • CI: existência de .github/workflows/, .gitlab-ci.yml, circleci/
  • Testes: presença de __tests__, *.test.ts, *_test.go, pytest.ini

03 / 10 kai-sdd
opt-in

Spec-Driven Development rígido. 8 fases com checkpoints obrigatórios e audit-trail.

O que faz

Impede o anti-pattern clássico de assistentes de IA: "vamos começar a codar". Força o squad a passar pelas fases de definição (spec, requirements, design) antes de gerar código, e por verificação/retro depois. Cada fase produz um artefato em docs/specs/_active/<slug>/.

Na prática — caso de uso real

  1. /kai-spec payments-gateway → cria diretório, abre spec.md com template
  2. /kai-spec-discuss → Claude faz perguntas até reduzir ambiguity score abaixo do threshold
  3. /kai-spec-requirements → escreve requirements.md (funcionais + não-funcionais + acceptance criteria)
  4. /kai-spec-design → escreve design.md (arquitetura, endpoints, modelos de dados, sequence diagrams em mermaid)
  5. /kai-spec-plan → escreve plan.md com tasks atômicas e ordem de execução
  6. /kai-spec-execute → executa o plano com checkpoints; cada task vira commit atômico
  7. /kai-spec-verify → verifica que tudo em requirements.md tem evidência (teste, demo, código)
  8. /kai-spec-retro → captura learnings, arquiva em _completed/

Tecnicamente

Estado da spec

Detectado pelo conjunto de arquivos presentes:

retro.md         → RETRO
verification.md  → VERIFY
execution.log    → EXECUTE
tasks.md         → PLAN
design.md        → DESIGN
requirements.md  → SPEC

Checkpoint enforcement

Antes de avançar para fase N+1, o comando confere que o artefato da fase N existe e foi assinado pelo squad (presença de seção "## Approved" no markdown).

04 / 10 kai-review
recomendado

Code review, security review e spec coverage como agentes formais. Cada um produz markdown estruturado.

O que faz

Substitui o reviewer humano por agentes especializados com mandatos diferentes. Não "olha o diff inteiro de forma genérica" — cada agente tem um framework próprio (OWASP para security, rubricas para code quality, comparação spec ↔ implementação para coverage).

Na prática

  1. Antes de abrir PR → /kai-review-pr → agente kai-code-reviewer roda passes de smell, complexidade, cobertura, naming, error handling.
  2. Se a feature toca crypto/auth/PII → /kai-review-security → kai-security-reviewer usa lista OWASP + threat model específico.
  3. Final do ciclo SDD → /kai-review-spec → kai-spec-reviewer confere se cada requirement em requirements.md tem evidência na implementação.
  4. Output dos três é markdown classificado: BLOCK / FLAG / PASS.
  5. /kai-review-fix aplica fixes propostos um a um, com confirmação.

Comandos (4)

/kai-review-pr/kai-review-security /kai-review-spec/kai-review-fix

09 / 10 kai-evals
opt-in

Eval framework para features de produção que usam LLM em runtime. Datasets golden, rubricas, A/B, CI.

O que faz

É o teste-suite das suas features de IA. Diferente de unit tests, evals lidam com saídas não-determinísticas: dataset golden de inputs + outputs esperados, rubricas multi-dimensional, runners que rodam o prompt N vezes e calculam métricas (accuracy, precision, faithfulness, latency).

Na prática

  1. Feature em prod usa LLM (ex.: classifier de tickets) → /kai-evals-init ticket-classifier cria scaffolding.
  2. Você popula evals/<feature>/dataset.jsonl com 30+ exemplos labeled.
  3. Define rubricas em rubrics.md (determinística: exact match; LLM-as-judge: faithfulness; custom: regex).
  4. /kai-evals-run ticket-classifier → roda o prompt atual contra o dataset, gera results/<timestamp>.json.
  5. Antes de trocar o prompt → /kai-evals-compare run-A run-B mostra delta em cada rubrica.
  6. No CI → /kai-evals-ci retorna exit 0/1 conforme threshold.

Estrutura de uma eval

evals/ticket-classifier/ ├── dataset.jsonl ← golden examples: input + expected ├── rubrics.md ← dimensões de avaliação ├── prompts/v1.txt ← prompts versionados ├── prompts/v2.txt ├── runs/ ← outputs históricos │ ├── 2026-05-10-v1.json │ └── 2026-05-13-v2.json └── ci-config.yml ← thresholds para gate de PR

3.2 Search BM25

Busca lexical com ranking BM25 puro shell sobre todo o conteúdo textual do projeto. Interface CLI estável para futuro upgrade a embeddings (v2.5+) sem breaking change.

Algoritmo

BM25 com k1=1.5, b=0.75. Tokenização: lowercase + split em não-alfanuméricos + remoção de stop-words PT-BR mínimo. IDF computado uma vez por build; TF por documento.

score(D, Q) = Σ IDF(qi) · (f(qi,D) · (k1+1)) / (f(qi,D) + k1 · (1 - b + b · |D|/avgdl))

onde:
  f(qi,D)  = frequência do termo qi no documento D
  |D|      = comprimento de D (em tokens)
  avgdl    = média de comprimento dos documentos
  IDF(qi)  = log((N - df + 0.5) / (df + 0.5) + 1)
  N        = total de documentos
  df       = quantos documentos contêm o termo
Cobertura
  • .kai/intel/graph.json — nós do knowledge graph
  • docs/specs/_active/**/*.md + _completed/**/*.md
  • docs/playbooks/**/*.md
  • docs/runbooks/**/*.md (se existir)
  • .kai/audit/*.log (somente com flag --audit)
Schema do índice
{
  "docs": [
    { "path": "docs/specs/_active/feat/design.md",
      "tf": { "retry": 4, "timeout": 2, "grpc": 1 },
      "dl": 482 },

      "tf": { "grpc": 2, "payments": 1 },

5. Sistema de hooks

Hooks são scripts shell executados em eventos do Claude Code. Recebem JSON via stdin, respondem por exit code + stdout. São a "imune system" do harness.

5.1 Categorias

CategoriaComportamentoExemplos
SEGURANÇASempre bloqueia. Sem exceção.secret-scan, pii-scan, private-key-scan
COMPLIANCEBloqueia. Exceção via /kai-exception (auditada).mcp-allowlist, pre-write-guard
PROCESSOWarn inicial. Vira block após maturidade.destructive-confirm
QUALIDADESempre warn. Não interrompe.session-start-banner, statusline

5.2 Eventos cobertos

EventoQuando disparaPlugins que usam
SessionStartAbertura da sessãokai-ai-core (banner)
PreToolUseAntes de cada tool callkai-ai-core, kai-security

5.3 Anatomia de um hook

Exemplo: plugins/kai-cost/hooks/scripts/cost-finalize.sh

#!/usr/bin/env bash
set -eo pipefail

# 1. Recebe payload via stdin
INPUT="$(cat 2>/dev/null || echo '{}')"

# 2. Extrai transcript_path (Claude Code Stop/SessionEnd)
TRANSCRIPT=$(echo "$INPUT" | jq -r '.transcript_path // empty')
[[ -z "$TRANSCRIPT" || ! -f "$TRANSCRIPT" ]] && exit 0

# 3. Agrega usage por modelo via jq
USAGE_AGG=$(jq -s '
  [.[]
   | select(.message.usage? // .usage?)
   | (.message.usage // .usage) as $u
   | { model: (.message.model // .model // "unknown"),
       in:  ($u.input_tokens // $u.prompt_tokens // 0),
       out: ($u.output_tokens // $u.completion_tokens // 0) }
  ]
  | group_by(.model)
  | map({ model: .[0].model,
          in:  (map(.in)  | add // 0),
          out: (map(.out) | add // 0) })
' "$TRANSCRIPT")

# 4. Append em .kai/audit/cost.log (pipe-separated)
echo "$USAGE_AGG" | jq -c '.[]' | while IFS= read -r row; do
  printf '%s | feature=%s | ... | source=transcript\n' "$NOW" "$FEATURE" \
    >> .kai/audit/cost.log
done

exit 0
